# 자기소개서

## 1. 구현 난이도가 가장 높았던 프로젝트

카카오 IDC 장애 발생 시 수동으로 트래픽을 전환해야 했고, 그 과정에서 7~8분이 소요되어 발송 실패율이 50%에 달했습니다. 전용선으로 IDC별 연결이 고정된 환경이라 인프라 레벨 Failover가 불가능했고, 애플리케이션에서 직접 해결해야 하는 구조였습니다.

가장 어려웠던 부분은 임계값 설계였습니다. 서비스 특성상 시간대별 트래픽 편차가 커서 고정 임계값으로는 정상 상황에서도 오탐이 잦았습니다. 이를 해결하기 위해 Resilience4j Sliding Window 기반으로 실패율을 실시간 집계하고 시간대별 동적 임계값을 적용했습니다. 각 IDC 상태는 Redis를 통해 인스턴스 간 공유하도록 했고, 상태 변경 시 RabbitMQ로 이벤트를 브로드캐스트해 다수 모듈이 동시에 전환되도록 구현했습니다.

결과적으로 10~30초 이내 자동 Failover를 달성했고, 단일 IDC 장애 시 발송 실패율을 50%에서 약 5%로 줄였습니다.

## 2. 가장 효과적으로 트러블슈팅했던 사례

운영 중 모니터링을 통해 Tomcat 워커 스레드의 CPU 사용률이 비정상적으로 급등하는 현상을 직접 발견했습니다. 피크 시 CPU 100%까지 치솟았지만 로그에는 별다른 에러가 없어 원인 파악이 어려운 상황이었습니다.

Java Thread Dump를 분석한 결과, BCrypt 암호화 작업에 의해 다수 Thread들이 CPU를 장시간 점유하고 있는 것을 확인했습니다. BCrypt는 무작위 대입 공격 방지를 위해 의도적으로 수만 번의 해시 연산을 수행하는 구조로, 트래픽이 몰릴수록 CPU 부하가 폭발적으로 증가했습니다.

대안으로 SHA-256을 검토했으나 API Key 엔트로피가 낮아 무작위 대입 공격에 취약하다고 판단해 서버키 기반 HMAC을 선택했습니다. 최초 인증은 BCrypt를 유지해 클라이언트 코드 수정 없이, 인증 성공 후 HMAC으로 서명한 값을 Redis에 저장해 이후 요청은 HMAC 인증으로 전환했습니다. 결과적으로 CPU 사용률이 약 76% 감소했습니다.
